## Analysis & Proposed Architecture (Context for Claude/Future Devs):
Побудова власної системи ботів та API всередині Knitly (подібно до Telegram) може стати тією самою "кілер-фічею", яка відрізнить платформу від усіх інших музичних та соціальних платформ. 

### Архітектура системи ботів Knitly:

1. **Knitly Bot API (Окремий сервіс/роут)**
   Замість того, щоб боти були просто користувачами, вони матимуть окремий тип акаунта `type: 'bot'` у базі даних (Firestore). Для них буде створено окремий набір HTTP/REST ендпоінтів (наприклад, через Firebase Cloud Functions).

2. **Knitly BotFather**
   Спеціальний системний бот, який використовуватиметься для реєстрації власних ботів. 
   Робс чи будь-хто інший зможе зайти в месенджер, написати Knitly BotFather команду `/newbot`, і система автоматично:
   - Створить документ бота.
   - Згенерує унікальний `API_TOKEN`.
   - Дозволить поставити йому ім'я, аватар та опис.

3. **Синхронізація (Допис ↔ Чат) / Канали**
   Створення сутності "Офіційна Сторінка" (або "Канал").
   - У неї є публічна сторінка (`/knitly_news`), яка виглядає як стандартний профіль.
   - Але також є "Канал" у месенджері, де користувачі можуть підписатись.
   - Коли адмін (або бот) публікує пост на `/knitly_news`, тригер Cloud Function бере об'єкт поста (текст + прикріплений трек + картинки) і автоматично розсилає його всім підписникам у месенджер по їхньому `chatId`.

### Чому це вигідно для Knitly:
- **Автоматизація:** Музичні куратори зможуть створювати ботів, які щоранку надсилають їхнім підписникам свіжі "трендові" треки.
- **Екосистема:** Розробники почнуть створювати міні-додатки (ігри, вікторини про артистів) прямо всередині Knitly.
- **Офіційні Канали:** Наявність каналу "Knitly News ✅", де всі користувачі автоматично будуть отримувати сповіщення про нові фічі прямо в зручному вбудованому месенджері, а не через email.
